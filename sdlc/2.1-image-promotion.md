# Lab 2.1 – Image Promotion

**Duration:** ~45 minutes  
**Prerequisites:** Completed Lab 1.4

## Learning Objectives

By the end of this lab you will be able to:
- Describe what image promotion means and how it differs from a new deployment
- Explain how semantic versioning signals potential breaking changes between major versions
- Use regression tests as a promotion gate to catch breaking changes before they reach staging or production
- Diagnose a failing test by reading DockerHub release notes rather than guessing from code
- Create a version-specific test suite without breaking existing tests for environments still on the previous version

---

## Promotion vs Deployment

Deploying a new version means building and shipping new code. Promoting a version means taking an image tag that has already been tested in one environment and moving it to the next. The image does not change — only the environment it runs in does. This distinction matters because promotion is deliberate and controlled: someone has decided that a version is ready to advance.

In the self-service pipeline pattern established in Lab 1.2, promotion is triggered manually. You run the pipeline for the target environment, supply the image tag, and the pipeline handles the rest. Nothing moves forward unless someone explicitly chooses to promote it.

---

## Semantic Versioning and Breaking Changes

A version number communicates intent. Under semantic versioning, a major version bump — from 1.x to 2.x — signals that the API contract has changed in a way that is not backward compatible. Something that worked against v1.x may not work against v2.x. This is expected during active development, but it means a major version promotion requires more than updating the tag. It requires verification.

Regression tests are the first line of defence. Running the test suite against a freshly promoted environment tells you immediately whether the new version behaves the way your existing integration code expects. If a test fails, that is the pipeline doing its job — stopping a breaking change from advancing further.

---

## Release Notes as a Starting Point

When a test fails after a version bump, the failure output tells you what the test observed. What it does not tell you is why the behaviour changed. That information lives in the release notes.

DockerHub image pages often include a description or changelog for each version. Reading the release notes for a major version before you start tells you exactly what to look for and avoids the longer loop of inspecting code, rerunning tests, and guessing. In practice, reading the changelog before promoting a major version prevents surprises.

## Hands On

In this exercise you will promote the Order Match API from v1.x to v2.0.0, observe a failing regression test, diagnose the breaking change using the DockerHub release notes, and fix the integration code so the full test suite passes.

1. **Promote v2.0.0 to dev.**

   In Bitbucket, navigate to **Pipelines → Run pipeline**. Select the `deploy-dev` custom pipeline. Set `TAG` to `v2.0.0` and run it. Wait for the pipeline to complete.

   After deployment, ECS Fargate assigns a new public IP to the replacement task. Find it in the AWS Console under **ECS → Clusters → order-match-api → Services → order-match-api-dev → Tasks** and copy the new IP.

2. **Run regression tests.**

   Run the `regression-tests` custom pipeline with `BASE_URL=http://<new-dev-ip>:8000` and `STAGE=dev`. Let it complete.

   One or more tests will fail. This is expected.

3. **Read the failure output.**

   Open the failed pipeline run. In the test output, find the assertion error for each failing test. Note exactly what the test asserted and what the API actually returned. Write it down — this is your diagnostic baseline.

4. **Read the DockerHub release notes.**

   Navigate to `hub.docker.com/r/innovationinsoftware/order-match-api`. Find the tags or description section for v2.x and read what changed. The release notes explain what is different about the API contract in this version.

   Do not start editing code until you have read the release notes. The changelog tells you the intent behind the change — the test failure only tells you the symptom.

5. **Create a v2 test suite.**

   First, rename `tests/test_api.py` to `tests/test_api_v1.py` so the naming is consistent with the `VERSION` variable you are about to add. Then create `tests/test_api_v2.py` with assertions updated to match the v2.x API contract you identified from the release notes. If needed, create `tests/seed_v2.py` as well rather than modifying the original.

   Then update the `regression-tests` custom pipeline in `bitbucket-pipelines.yml` to accept a `VERSION` variable and run the appropriate file. Add `VERSION` to the pipeline's `variables` list with a default of `v1`, and change the pytest command to use it:

   ```yaml
   - step:
       script:
         - pip install -r requirements.txt
         - pytest tests/test_api_${VERSION}.py --junitxml=test-results/results.xml --junit-prefix=${STAGE} -v
   ```

   Commit and push.

6. **Confirm all tests pass.**

   Run `regression-tests` with `BASE_URL=http://<dev-ip>:8000`, `STAGE=dev`, and `VERSION=v2`. All tests should pass. Review the **Tests** tab — the history now shows the earlier v1 failure run, then the current v2 passing run. The original `test_api.py` is still intact and will be used whenever `VERSION=v1` is selected.

7. **Challenge: promote to QA and prod.**

   Promote v2.0.0 to QA using the `deploy-qa` pipeline, then run `regression-tests` with the QA task's public IP, `STAGE=qa`, and `VERSION=v2`. Once QA passes, promote to prod using `deploy-prod` and run `regression-tests` one final time with `STAGE=prod` and `VERSION=v2`. All tests should pass without further code changes. Confirm all three environments are running v2.0.0 before considering the promotion complete.

## Summary

In this lab you:
- Promoted the Order Match API from v1.x to v2.0.0 using the existing self-service pipeline
- Used regression tests as a promotion gate and observed a breaking change surface as a test failure
- Diagnosed the change by reading DockerHub release notes rather than guessing from the code
- Created `tests/test_api_v2.py` to cover the v2.x API contract without modifying the existing v1 tests
- Added a `VERSION` variable to the `regression-tests` pipeline so the right suite runs for each environment
- Confirmed all tests passed across environments before advancing the version further
