---

## The Weekly Test Cycle

In teams that ship regularly, testing is not ad hoc — it follows a repeatable cadence. A typical weekly cycle starts by seeding the environment with a fresh synthetic dataset, deploying the target version, running regression tests, and reviewing results. The following week the cycle begins again. This structure means the team always starts from a known, clean state and always ends with a documented record of what passed and what failed.

---

## Regression Testing

A regression is when a new change breaks something that previously worked. It happens frequently in active development: a feature added to one endpoint unexpectedly alters the behavior of another, or a dependency upgrade changes how responses are serialised. Without a fixed set of checks, regressions surface in production instead of in a test environment.

Regression tests are assertions run against a deployed service to confirm it still behaves as expected. The same tests run every cycle, every week. If a test that passed last week fails this week, the pipeline makes that visible before any user is affected.

---

## Test Reporting in Bitbucket

Bitbucket Pipelines reads JUnit XML natively. pytest generates this format with a single flag: `--junitxml=<path>`. When the pipeline step declares that file as a report artifact, Bitbucket parses it and displays a pass percentage and a table of individual test results on the pipeline run. Each subsequent run is recorded, so you can scroll back through the table and see exactly when a test started failing.

The report is captured even if some tests fail, so you can always inspect results regardless of whether the overall pipeline step succeeded.

## Hands On

In this exercise you will write regression tests for the Order Match API, add a test pipeline step, run it against your dev environment, and then execute the complete weekly cycle end to end.

1. **Create the test files.**

   In the `tests/` directory of your forked repo (where `seed.py` already lives), add two new files.

   `tests/requirements.txt`:

   ```
   pytest==8.3.0
   requests==2.32.0
   ```

   `tests/test_api.py`:

   ```python
   import os
   import uuid
   import requests

   BASE_URL = os.environ.get("BASE_URL", "http://localhost:8000")


   def test_get_orders():
       r = requests.get(f"{BASE_URL}/orders")
       assert r.status_code == 200
       assert isinstance(r.json(), list)


   def test_post_order():
       body = {"id": str(uuid.uuid4()), "side": "buy", "price": 10, "quantity": 5, "status": "pending"}
       r = requests.post(f"{BASE_URL}/orders", json=body)
       assert r.status_code == 201
       assert "id" in r.json()


   def test_get_order_by_id():
       body = {"id": str(uuid.uuid4()), "side": "sell", "price": 12, "quantity": 3, "status": "pending"}
       r = requests.post(f"{BASE_URL}/orders", json=body)
       order_id = r.json()["id"]
       r2 = requests.get(f"{BASE_URL}/orders/{order_id}")
       assert r2.status_code == 200
       assert r2.json()["id"] == order_id


   def test_get_queue():
       r = requests.get(f"{BASE_URL}/queue")
       assert r.status_code == 200
       assert isinstance(r.json(), list)


   def test_order_not_found():
       r = requests.get(f"{BASE_URL}/orders/999999")
       assert r.status_code == 404
   ```

   Five tests cover every endpoint: listing orders, creating an order, retrieving an order by its ID, reading the match queue, and confirming that a non-existent ID returns 404. The `test_get_order_by_id` test creates an order first and then retrieves it — this is a round-trip test that validates both the write and read paths together.

2. **Run the tests locally (optional).**

   ```
   pip install -r tests/requirements.txt
   BASE_URL=http://<dev-ip>:8000 pytest tests/ -v
   ```

   Each test name is printed with PASS or FAIL, followed by a summary line showing the count and total duration. This is identical output to what the pipeline will produce.

3. **Add the test pipeline step.**

   Open `bitbucket-pipelines.yml` and add a `regression-tests` custom pipeline after the existing steps:

   ```yaml
   regression-tests:
     - variables:
         - name: BASE_URL
           description: "Base URL of the target environment (e.g. http://<public-ip>:8000)"
         - name: STAGE
           description: "Environment name used to label results (e.g. dev, qa, prod)"
     - step:
         name: Regression Tests
         image: python:3.11-slim
         script:
           - pip install -r tests/requirements.txt -q
           - mkdir -p test-results
           - pytest tests/ --junitxml=test-results/results.xml --junit-prefix=$STAGE -v
         artifacts:
           reports:
             junit: test-results/results.xml
   ```

   - `mkdir -p test-results` ensures the output directory exists — pytest will not create it, and without it no XML is written and Bitbucket has nothing to parse.
   - `--junitxml` writes results in the standard XML format.
   - `--junit-prefix=$STAGE` prefixes every test name with the environment name, so runs against dev and qa are distinguishable in the history table.
   - `artifacts: reports: junit:` tells Bitbucket to parse that file and render the Tests tab. The report is captured even when tests fail.

4. **Commit and push.**

   Stage `tests/requirements.txt`, `tests/test_api.py`, and the updated `bitbucket-pipelines.yml`. Push to your repo.

5. **Run the test pipeline.**

   In Bitbucket, navigate to **Pipelines → Run pipeline**. Select the `regression-tests` custom pipeline. Enter the dev service's public IP as `BASE_URL` (e.g., `http://1.2.3.4:8000`) and `dev` as `STAGE`. Run it.

6. **View test results.**

   When the pipeline finishes, navigate to **Tests** in the repository menu. You will see a pass percentage and a table listing each test with its pass/fail status and duration. Run the pipeline a second time — both runs now appear in the history table, which is how the team tracks regressions week over week.

7. **Challenge.**

   **Part 1 — Fix the failing test.** Run `regression-tests`. One test will fail. Open **Tests** in the repository menu, read the actual status code returned by the API, then check `/docs` on your dev environment to confirm the correct value. Fix the assertion in `tests/test_api.py`, push, and rerun until all five tests pass.

   **Part 2 — Add latency coverage.** Add the following test to `tests/test_api.py`:

   ```python
   import time

   def test_bulk_order_latency():
       start = time.time()
       for _ in range(10):
           body = {"id": str(uuid.uuid4()), "side": "buy", "price": 10, "quantity": 1, "status": "pending"}
           r = requests.post(f"{BASE_URL}/orders", json=body)
           r.raise_for_status()
       elapsed = time.time() - start
       assert elapsed < 5.0, f"10 orders took {elapsed:.2f}s — expected under 5s"
   ```

   This submits 10 orders in sequence and asserts the total round-trip time stays under 5 seconds. A latency regression — caused by a slow dependency, a missing index, or degraded infrastructure — will surface here before it reaches production. Push and confirm all six tests pass.

   **Part 3 — Run across environments.** Run `regression-tests` twice: once with `STAGE=dev` and your dev IP, once with `STAGE=qa` and your QA IP. Open **Tests** in the repository menu and confirm the history table shows results prefixed by environment name, making it clear which run targeted which stage.
