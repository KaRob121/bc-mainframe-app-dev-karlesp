# Lab 2.3 – Migration Planning and Vendor Tooling

**Duration:** ~25 minutes  
**Prerequisites:** Completed Labs 2.1 and 2.2

## Learning Objectives

By the end of this lab you will be able to:
- Analyze your weekly delivery cycle and identify which operations would change on a different platform
- Distinguish between platform-specific mechanisms and platform-agnostic logic
- Produce a migration assessment that clarifies what changes, what stays the same, and where risk lives
- Sketch a parallel-run strategy to validate a platform migration safely

---

## Why This Matters

You have built a complete weekly cycle over the past two days: refresh environments on schedule, seed them with test data, deploy code through promotion gates, and validate with regression tests. These are not ECS-specific or Bitbucket-specific patterns — they are delivery lifecycle operations. Any platform has mechanisms for them, but they do not all work the same way.

Before a team migrates to a new platform, it must answer a hard question: **What changes operationally, and what stays the same?** Some operations may require new tooling or a different trigger mechanism. Others (your test scripts, for example) may not change at all. The gaps between the two systems — what has to be done differently — are where migration risk lives. Understanding those gaps is how you plan for change.

In this lab, you will analyze the full weekly cycle you built and identify exactly what would change if you ran it on Kubernetes instead of ECS.

---

## Thinking Through the Migration

Use these questions to frame your analysis:

**About the weekly cycle you built:**
- How does environment isolation work in ECS? How would it work in Kubernetes?
- How does the refresh happen today? Would the mechanism change?
- How is seed data applied in your current setup? Would that be different?
- How are promotion gates enforced now? What would enforce them in Kubernetes?
- How do regression tests run today? Would they need to change?

**About what stays the same:**
- Which parts of the weekly cycle are *pure operations* and would not change regardless of platform?
- Which of your scripts or logic are platform-agnostic?

**About migration risk:**
- If one operation changes (e.g., how you trigger deployment), what cascades from that?
- How would you know the migration succeeded? What is your primary validation?
- What could go wrong, and how would you detect it?

---

## Vendor Evaluation Framework

Do not choose a vendor based on feature lists alone. Evaluate each option against the delivery operations you already run.

Use the following prompts during research:

**Kubernetes control plane options (for example EKS, AKS, on-premises):**
- How is the control plane operated and upgraded?
- What is the expected operational overhead for your team?
- How does it support your environment model (dev, qa, prod)?
- What is the rollback path when a platform upgrade causes instability?

**Cluster management and delivery options (for example ArgoCD, OpenShift, or others):**
- How are deployment promotion, drift detection, and rollback handled?
- How are credentials and secrets managed across environments?
- How are job dependencies represented and validated?
- What observability exists for sync failures, job failures, and policy violations?

Your output in this lab is not a final vendor decision. It is a short, evidence-based comparison that identifies what you still need to validate.

## Hands On

1. **Write your migration assessment.**

   Create `docs/k8s-migration-assessment.md` in your forked repo. Structure it as follows and fill in each section based on the weekly cycle you built over the past two days:

   ```markdown
   # Kubernetes Migration Assessment – Order Match API

   ## Our Current Weekly Cycle
   Summarize what you built: environment structure, how refresh is triggered, how seeding works, 
   how promotion happens, how tests are run. Focus on the *operations*, not the underlying AWS infrastructure.

   ## What Changes if We Migrate to Kubernetes
   For each operation in your weekly cycle, identify whether it changes and how. 
   Be specific about what is different and why.

   ## What Stays the Same
   Which parts of your weekly cycle are independent of the platform? 
   Which scripts or logic would not change?

   ## How We'd Know It Worked
   What is your primary validation that the migration succeeded? 
   How would you use your regression tests to confirm the platform migration is sound?

   ## Migration Risks
   List three things that could go wrong during migration and how you would detect each one.

   ## Vendor Landscape
   Research and briefly describe your options for:
   - Kubernetes control plane (EKS, AKS, or on-premises alternatives)
   - Cluster management and delivery tooling (options include ArgoCD, OpenShift, or others)
   Do not implement; just identify what is available and note what you would evaluate before deciding.
   ```

   Commit the file to your repo.

2. **Challenge: plan a safe migration.**

   Add a **Parallel Run Strategy** section to your assessment. Describe how you would run ECS and Kubernetes side by side for one to two weeks to reduce risk. Include:
   - How both systems receive the same regression test runs (hint: use the `STAGE` variable you added in Lab 2.1)
   - What "passing" looks like across both systems
   - When you would decommission ECS and what would trigger a rollback to ECS if something fails
   
   This is a change management exercise — you are planning how to validate a major platform change without exposing the business to outage risk.

3. **Add a migration runbook outline.**

   Add a **Migration Runbook** section to your assessment that captures execution order at a practical level. Keep it concise.

   Include at least these checkpoints:
   - Pre-migration entry criteria (current platform stable, tests passing, known baseline)
   - Platform readiness (cluster, namespaces, access, and core manifests in place)
   - Dependency sequencing (deploy first, seed second, test third)
   - Validation gates per environment (dev, qa, prod)
   - Cutover criteria and rollback triggers

   This section should read like a step-by-step migration plan you could hand to an operations team for execution rehearsal.

In this lab you:
- Analyzed the full weekly delivery cycle you built and identified which operations change on a new platform
- Distinguished between platform-specific mechanisms and platform-agnostic operations
