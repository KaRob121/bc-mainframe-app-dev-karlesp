## From ECS to Kubernetes

Your org want to migrate too Kubernetes from ECS. ECS and Kubernetes both run containers, but the abstraction models are different. In ECS, a task definition is a blueprint for a container, an ECS service keeps a desired number of tasks running, and each Fargate task gets a public IP directly. In Kubernetes, a Pod is the smallest deployable unit, a Deployment manages the desired number of Pod replicas, and a Service provides a stable network endpoint in front of them. Traffic reaches the application through a Service — not through a raw Pod IP.

| ECS | Kubernetes |
|---|---|
| Task Definition | Pod spec (inside a Deployment) |
| ECS Service | Deployment + Service |
| Public IP on Fargate task | LoadBalancer Service (AWS ELB) |
| CloudWatch Log Group | `kubectl logs <pod-name>` |
| IAM execution role | ServiceAccount |
| `aws ecs update-service` | ArgoCD sync (GitOps) |
| ECS Cluster | Kubernetes Namespace |

## eksctl and AWS CloudShell

`eksctl` is a CLI tool that creates and manages EKS clusters. A single command provisions the EKS control plane, managed node group, VPC, subnets, IAM roles, and kubeconfig. AWS CloudShell is a browser-based terminal inside the AWS Console that comes with the AWS CLI already configured against your account. Running eksctl from CloudShell means no local installation is required and the cluster credentials stay in your session automatically.

---

## GitOps with ArgoCD

ArgoCD is a tool we will install that runs inside the cluster and continuously pulls the desired state from your git repository. When you commit a change to a manifest, ArgoCD detects it and reconciles the cluster to match. If someone manually changes the cluster, ArgoCD reverts the drift. The repository becomes the single source of truth.

---

## Handling Dependencies in Jobs and Datasets

Kubernetes makes dependencies explicit. A seed Job depends on both application readiness and dataset configuration being present. If either dependency is missing, the job fails and your weekly cycle breaks.

Use this dependency chain when reasoning about delivery flow:
- Deployment and Service are healthy and reachable
- Dataset configuration exists (ConfigMap/Secret values are correct)
- Seed Job runs successfully and exits cleanly
- Regression tests run against seeded data

In practice, this means you should validate dependencies before running seed operations:
- Confirm Deployment readiness
- Confirm Service endpoint exists and responds
- Confirm ConfigMap and Secret resources are present before Job execution

When you document migration plans in Lab 2.3, include these dependencies explicitly. This is where many "the code deployed but tests failed" incidents originate.

## Hands On

In this exercise you will create an EKS cluster from CloudShell, prepare your Kubernetes manifests in Bitbucket, install ArgoCD, and let it deploy and manage the Order Match API automatically. The cluster takes approximately 15–20 minutes to provision. You will start that process first and use the wait time productively.


1. **Open CloudShell and install the tools.**

   In the AWS Console, click the CloudShell icon in the top navigation bar (the terminal prompt icon). A browser-based terminal opens.

   Install kubectl and eksctl:

   ```bash
   mkdir -p $HOME/.local/bin && cd $HOME/.local/bin

   curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
   chmod +x kubectl

   curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
   mv /tmp/eksctl $HOME/.local/bin
   cd ~
   ```

   ```bash
   kubectl version --client
   eksctl version
   ```

2. **Generate an SSH key pair and upload it to AWS.**

   ```bash
   ssh-keygen -t rsa -b 2048 -f ~/.ssh/eks-key
   aws ec2 import-key-pair \
     --region us-east-1 \
     --key-name eks-key \
     --public-key-material fileb://~/.ssh/eks-key.pub
   ```

   This key provides optional SSH access to EC2 worker nodes for debugging.

3. **Create the EKS cluster.**

   Run the following command. This takes approximately 15–20 minutes — proceed to the next step while it runs:

   ```bash
   eksctl create cluster \
     --name order-match-api \
     --region us-east-1 \
     --nodes 2 \
     --node-type t3.medium \
     --with-oidc \
     --ssh-access \
     --ssh-public-key eks-key \
     --managed
   ```

   `--with-oidc` enables IAM roles for service accounts. `--managed` creates an AWS-managed node group that handles node patching automatically. eksctl writes the kubeconfig to your CloudShell session automatically when provisioning completes.

4. **Create the Kubernetes manifests while the cluster provisions.**

   Open your forked Bitbucket repository in a browser. Use the web editor (**Source → Add file**) to create `k8s/deployment.yaml`:

   ```yaml
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: order-match-api
     namespace: dev
   spec:
     replicas: 1
     selector:
       matchLabels:
         app: order-match-api
     template:
       metadata:
         labels:
           app: order-match-api
       spec:
         containers:
           - name: order-match-api
             image: docker.io/innovationinsoftware/order-match-api:v2.0.0
             ports:
               - containerPort: 8000
   ---
   apiVersion: v1
   kind: Service
   metadata:
     name: order-match-api
     namespace: dev
   spec:
     type: LoadBalancer
     selector:
       app: order-match-api
     ports:
       - port: 8000
         targetPort: 8000
   ```

   Commit directly to main. You will add the seed Job manifest later, once you have the LoadBalancer hostname.

  While still in Bitbucket, go to **Settings → Security → Access tokens** and create a token for repository access. Select **Repository: Read and write** and copy the generated secret. ArgoCD will use this token to read the repository.

5. **Verify the cluster and install ArgoCD.**

   Back in CloudShell, confirm the cluster is ready:

   ```bash
   kubectl get nodes
   ```

   Both nodes should show status `Ready`. Create the dev namespace:

   ```bash
   kubectl create namespace dev
   ```

   Install ArgoCD:

   ```bash
   kubectl create namespace argocd
   kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
   kubectl wait --for=condition=available --timeout=180s deployment/argocd-server -n argocd
   ```

   Expose the ArgoCD server with a public Load Balancer:

   ```bash
   kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "LoadBalancer"}}'
   kubectl get svc argocd-server -n argocd -w
   ```

   Wait for an external hostname to appear in the `EXTERNAL-IP` column, then press Ctrl+C. Copy it.

6. **Log in to the ArgoCD UI.**

   Retrieve the initial admin password:

   ```bash
   kubectl -n argocd get secret argocd-initial-admin-secret \
     -o jsonpath="{.data.password}" | base64 -d && echo
   ```

   Open `https://<argocd-lb-hostname>` in a browser. If it does not work, wait a few minutes for DNS.
   
   When your browser warns that the connection is not valid, accept the self-signed certificate warning. Log in with username `admin` and the password above.

7. **Connect your Bitbucket repository and create the Application.**

   In the ArgoCD UI, go to **Settings → Repositories → Connect Repo** and enter:

   - Connection method: HTTPS
   - Repository URL: `https://bitbucket.org/<your-username>/<your-repo-name>`
   - Username: x-token-auth
  - Password: the read-only credential from step 4

   Click **Connect**. The repository should show a green connected status.

   Now create the Application. Click **Applications → New App** and fill in:

   | Field | Value |
   |---|---|
   | Application Name | `order-match-api` |
   | Project | `default` |
   | Sync Policy | Automatic — enable **Prune Resources** and **Self Heal** |
   | Repository URL | select your connected repo |
   | Revision | `HEAD` |
   | Path | `k8s` |
   | Cluster URL | `https://kubernetes.default.svc` |
   | Namespace | `dev` |

   Click **Create**. ArgoCD immediately begins syncing.

8. **Watch the sync and get the application URL.**

   In the ArgoCD UI, click into the `order-match-api` Application. A graph shows the Deployment and Service being created. Wait for both to show green (Synced/Healthy).

   Back in CloudShell, wait for the application's Load Balancer hostname:

   ```bash
   kubectl get service order-match-api -n dev -w
   ```

  Copy the `EXTERNAL-IP` hostname once it appears. Navigate to `http://<app-lb-hostname>:8000/docs` in a browser and confirm the Swagger UI loads.

  You will use this external hostname for browser checks and Bitbucket-hosted regression tests. For in-cluster jobs, prefer the stable Kubernetes Service DNS name.

9. **Add the seed Job.**

  Open the Bitbucket web editor and create `k8s/seed-job.yaml`. Use the in-cluster Service DNS name so this works the same way for everyone:

   ```yaml
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: seed-config
     namespace: dev
   data:
     api_url: http://order-match-api.dev.svc.cluster.local:8000
   ---
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: seed-script
     namespace: dev
   data:
     seed.py: |
       import os, random, uuid, requests
       url = os.environ["BASE_URL"]
       for _ in range(20):
           body = {
               "id": str(uuid.uuid4()),
           "symbol": random.choice(["AAPL", "MSFT", "GOOG"]),
               "side": random.choice(["buy", "sell"]),
               "price": random.randint(50, 200),
               "quantity": random.randint(1, 100),
               "status": "pending",
           }
           r = requests.post(f"{url}/orders", json=body)
           r.raise_for_status()
       print(f"Seeded 20 orders to {url}")
   ---
   apiVersion: batch/v1
   kind: Job
   metadata:
     name: seed-dev
     namespace: dev
   spec:
     template:
       spec:
         restartPolicy: Never
         volumes:
           - name: seed-script
             configMap:
               name: seed-script
         containers:
           - name: seeder
             image: python:3.11-slim
             env:
               - name: BASE_URL
                 valueFrom:
                   configMapKeyRef:
                     name: seed-config
                     key: api_url
             volumeMounts:
               - name: seed-script
                 mountPath: /scripts
             command:
               - bash
               - -c
               - pip install requests -q && python /scripts/seed.py
   ```

   Commit to main. ArgoCD detects the new file and syncs automatically — watch it appear in the Application graph. Once the Job completes, verify in CloudShell:

   ```bash
   kubectl logs job/seed-dev -n dev
   ```

  **Note:** Kubernetes Jobs are one-shot by default. `seed-dev` runs once and stays in `Completed` state. To run it again, create a new Job instance (for example, delete `seed-dev` so ArgoCD recreates it, or commit a new Job name such as `seed-dev-rerun`).

   If you want automatic recurring seed runs, use a `CronJob` with a schedule. Example schedules:
   - Mon-Fri every 20 minutes: `*/20 * * * 1-5`
   - Every Friday night at 23:00: `0 23 * * 5`
   - Early Monday morning at 06:00: `0 6 * * 1`

   Example CronJob snippet:

   ```yaml
   apiVersion: batch/v1
   kind: CronJob
   metadata:
     name: seed-dev-scheduled
     namespace: dev
   spec:
     schedule: "*/20 * * * 1-5"
     jobTemplate:
       spec:
         template:
           spec:
             restartPolicy: Never
             containers:
               - name: seeder
                 image: python:3.11-slim
                 command: ["bash", "-c", "echo run seed script here"]
   ```

  Navigate to `http://<app-lb-hostname>:8000/orders` and confirm 20 synthetic orders appear.

10. **Run regression tests.**

  In Bitbucket, run the `regression-tests` custom pipeline with `BASE_URL=http://<app-lb-hostname>:8000`, `STAGE=eks-argocd`, and `VERSION=v2`. This uses the v2 test naming convention introduced in Lab 2.1 (for example `tests/test_api_v2.py`). All tests should pass against the EKS deployment.

    In Bitbucket, add a repository variable `EKS_DEV_URL` set to `http://<app-lb-hostname>:8000`. This is used in Lab 2.3.

11. **Challenge: demonstrate GitOps and drift detection.**

    **Part 1 — GitOps deployment.** In the Bitbucket web editor, open `k8s/deployment.yaml` and change `replicas: 1` to `replicas: 2`. Commit to main. In the ArgoCD UI, click **Refresh** or wait up to 3 minutes for the automatic poll. ArgoCD syncs and a second Pod appears in the graph. Change it back to `replicas: 1` and commit — ArgoCD scales back down. The push to git was the entire deployment action: no pipeline run, no kubectl command.

    **Part 2 — Drift detection.** In CloudShell, manually delete the Deployment:

    ```bash
    kubectl delete deployment order-match-api -n dev
    ```

    Watch the ArgoCD UI. With `selfHeal` enabled, ArgoCD detects the missing resource and recreates the Deployment within about 30 seconds. Note the time.

    To see the alternative, open the ArgoCD Application, click **App Details → Edit**, and disable **Self Heal**. Delete the Deployment again. This time ArgoCD shows the Application as `OutOfSync` but takes no action. Click **Sync** manually to restore it. This is the tradeoff: automated self-healing versus supervised reconciliation where a human approves changes before they are applied.
