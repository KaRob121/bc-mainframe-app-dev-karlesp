## Why This Matters

Data is only useful if someone acts on it. Dashboards translate raw platform data into visibility that helps stakeholders — product owners, project managers, operations leads — make faster, better-informed decisions.

As your team moves through a delivery lifecycle or migration, dashboards become the shared source of truth for release health, incident trends, and progress toward goals. Knowing how to build and read them is a core skill for anyone responsible for system reliability or software delivery.

---

## Background

### What Makes a Good Dashboard

Not all dashboards are useful. The best ones follow three principles:

| Principle | What It Means |
|---|---|
| **Simplicity** | Focus on one area of your ecosystem at a time — not everything at once |
| **Meaningful** | Show metrics that drive decisions, not just metrics that are easy to collect |
| **Accurate** | Data and calculations must be verifiable and explainable to stakeholders |

### Metrics

Metrics answer the questions stakeholders care about most:

- How often does our system go offline?
- How fast can we recover from an outage?
- How long will it take to release the next patch?

Before any project begins, the team should agree on which metrics matter. Stakeholders make decisions about staffing, budgets, timelines, and priorities based on the data you provide — so what you measure shapes what gets attention.

### SLOs

A **Service Level Objective (SLO)** puts a measurable goal on a system behavior. SLOs turn vague reliability expectations into trackable targets.

Example:
> _99% of level 2 incidents resolved within 1 week per month_

This SLO gives the team a clear target and gives stakeholders a way to evaluate performance objectively.

### Reporting and Stakeholder Communication

Dashboards offer real-time visibility, but stakeholders typically need more than a live link. Screenshots, slide summaries, and regular status updates give context to the numbers. Different roles care about different metrics:

- **Product Owners** — operational metrics: uptime, incident volume, resolution time
- **Project Managers** — delivery metrics: milestone completion, release cadence, open blockers
- **Operations Leads** — reliability metrics: MTTR, change success rate, rollback frequency

Be prepared to explain your data. Stakeholders will ask questions, and the numbers need to tell a coherent story.

## Hands On

### Part 1 - Explore Dashboards in ServiceNow

1. Log in to your ServiceNow developer instance from Lab 3.2.

2. In the top navigation, click **All** and search for **Dashboards**. Open the Dashboards section.

3. Find a dashboard related to **incidents or changes**. Look for metrics like:
   - **MTTR** (Mean Time To Resolve)
   - Incident volume by priority
   - Change request status breakdown

4. Note what each visual is showing. Ask yourself:
   - Is this dashboard simple and focused, or cluttered?
   - Would a stakeholder understand this without explanation?

---

### Part 2 - Create Activity to Generate Data

To see dashboards respond to real data, you will create some activity in the platform.

1. Navigate to **Incident > Create New** and create two or three test incidents:
   - Set different **Priority** levels (High, Medium, Low)
   - Add a short description and assignment group

2. Open your incidents from the list and add a **comment** to each one.

3. Mark at least one **high priority** incident as **Resolved**.

4. Navigate to **Change > Create New** and create one change request:
   - Fill in the short description and implementation plan fields
   - Set the state to **Assess**

5. Return to your dashboard from Part 1. Look for any visuals that reflect the activity you just created. What do they tell you about the state of the system?

---

### Part 3 - Explore Indicators and Reports

**Indicators** are tracked values that feed into dashboards. They represent key measurements over time.

1. Under Dashboards, open **Indicators**.

   > Note: Your trial instance may not have historical data, but review the indicator names available.

2. Look for indicators like:
   - `Average close time of changes` — useful for estimating release duration
   - `Percentage of changes implemented successfully`
   - `Number of open high-priority incidents`

3. Navigate to **All > Reports** and search for **open change requests**.

4. Open the report and review its contents. Notice the filters and grouping options.

5. Find an active change request in the report results. Open it and look for the **CAB (Change Advisory Board)** field. This is where a formal review board is assigned before a change goes to production.

6. Spend a few minutes exploring other available reports.

### Part 4 - Plan Two Dashboards

You will not build these dashboards in the platform, but you will design them. This is the kind of planning that happens before any dashboard is created in a real environment.

For each dashboard below, identify **at least three metrics** you would include and write one sentence explaining why each metric matters.

**Dashboard 1: Release Management Dashboard**

This dashboard helps stakeholders understand how the release process is performing.

Starter metrics to build on:
- Average time a change request is open before deployment
- Percentage of failed releases in the last 30 days

Add at least one more metric of your own and explain your reasoning.

---

**Dashboard 2: Migration Dashboard**

This dashboard tracks the status of a major platform migration. It should answer: _"Are we on track, and is the migration safe?"_

Starter metrics to build on:
- Percentage of passing tests in the new environment
- Error rate after migration
- Number of pipelines remaining to be created
- Last successful rollback test date

Add at least one more metric of your own and explain your reasoning.
