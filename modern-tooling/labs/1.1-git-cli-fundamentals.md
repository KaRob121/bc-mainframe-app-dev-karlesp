# Lab 1.1 – Git CLI Fundamentals

**Duration:** ~30 minutes  
**Prerequisites:** Git installed (Git Bash on Windows), a text editor

## Learning Objectives

By the end of this lab you will be able to:
- Initialize a Git repository and understand what the `.git` folder contains
- Stage and commit changes using the command line
- View commit history and compare file versions with `git diff`
- Create a branch, make changes on it, and merge it back to `main`

---

## Overview

Git is a distributed version control system that tracks changes to files by creating snapshots called commits. Every repository stores its full history in a hidden `.git` folder on your machine. You do not need a server or an internet connection to use Git. It works entirely locally.

This lab focuses on the command line. Many developers use VS Code or other GUI tools day-to-day, but understanding the CLI builds the mental model that makes every other tool easier to use. The goal is not mastery. It is to remove any fear of typing Git commands.

---

## Hands On

Open **Git Bash** (Windows) or your terminal (Mac/Linux) and follow the steps below.

### Part 1 – Initialize a Repository

1. Create a new folder for this exercise and navigate into it:

   ```bash
   mkdir git-lab
   cd git-lab
   ```

2. Initialize a Git repository:

   ```bash
   git init
   ```

   Git creates a hidden `.git` folder inside `git-lab`. This folder is the repository. It stores all commits, branch information, and configuration. You should never manually edit files inside `.git`.

3. Check the current state of the repository:

   ```bash
   git status
   ```

   Git reports that you are on branch `main` (or `master`) and there is nothing to commit yet.

---

### Part 2 – Make Your First Commit

4. Create a plain text file named `notes.txt`:

   ```bash
   echo "Version 1 - initial content" > notes.txt
   ```

5. Check status again. Git now reports `notes.txt` as an **untracked file**, meaning it exists on disk but Git is not yet watching it:

   ```bash
   git status
   ```

6. Stage the file. Staging tells Git which changes you want to include in the next commit:

   ```bash
   git add notes.txt
   ```

7. Commit the staged change. The `-m` flag attaches a message describing what changed and why:

   ```bash
   git commit -m "Add initial notes file"
   ```

8. View the commit history:

   ```bash
   git log
   ```

   Git displays the commit hash, author, date, and message. The hash is a unique identifier for this snapshot. You can use it to restore or compare versions at any time.

---

### Part 3 – Make a Second Commit and Compare Versions

9. Edit `notes.txt` to add a second line:

   ```bash
   echo "Version 2 - added more details" >> notes.txt
   ```

10. Before committing, compare the working copy against the last commit:

    ```bash
    git diff
    ```

    Lines beginning with `+` are additions. Lines beginning with `-` are removals. This is how Git shows you exactly what changed before you commit.

11. Stage and commit the update:

    ```bash
    git add notes.txt
    git commit -m "Update notes with additional details"
    ```

12. View the log again. You now have two commits:

    ```bash
    git log --oneline
    ```

    The `--oneline` flag condenses each commit to a single line, which is useful when the history is long.

---

### Part 4 – Create a Branch and Merge

13. Create a new branch named `feature/experiment`:

    ```bash
    git branch feature/experiment
    ```

14. Switch to the new branch:

    ```bash
    git checkout feature/experiment
    ```

    > You can combine these two steps with `git checkout -b feature/experiment`.

15. Add a new file on this branch:

    ```bash
    echo "This change is isolated to the feature branch" > experiment.txt
    git add experiment.txt
    git commit -m "Add experiment file on feature branch"
    ```

16. Switch back to `main` and confirm `experiment.txt` is not there:

    ```bash
    git checkout main
    ls
    ```

    The file only exists on the feature branch. This is the core value of branching. Changes are isolated and do not affect the stable branch until you are ready.

17. Merge the feature branch into `main`:

    ```bash
    git merge feature/experiment
    ```

    Git combines the changes. Both files now exist on `main`.

18. View the final log to see the complete history:

    ```bash
    git log --oneline
    ```

## Summary

In this lab you:
- Initialized a local Git repository and understood the role of the `.git` folder
- Staged and committed changes, building a commit history from scratch
- Used `git diff` to compare file versions before committing
- Created a branch, made isolated changes, and merged them back into `main`

## Challenge

Intentionally create a merge conflict. Here is how:

1. Create two branches from `main`: `branch-a` and `branch-b`
2. On each branch, edit the same line in `notes.txt` with different content and commit
3. Merge `branch-a` into `main`, then attempt to merge `branch-b`
4. Git will report a conflict. Open `notes.txt`, find the conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`), choose which version to keep, then stage and commit the resolution

This is the same process a team uses when two developers change the same code at the same time.
