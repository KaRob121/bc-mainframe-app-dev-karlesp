# Lab 1.2 – Learn Git Branching

**Duration:** ~30 minutes  
**Prerequisites:** A modern web browser

## Learning Objectives

By the end of this lab you will be able to:
- Visualize how commits, branches, and merges relate to each other
- Understand how rebasing differs from merging
- Navigate a repository history using relative references
- Explain the difference between local branches and remote tracking branches

---

## Overview

Reading about Git branching is one thing. Seeing it move is another. This lab uses an interactive browser-based simulator called **Learn Git Branching** that visualizes exactly what happens to the commit graph as you type each command.

Every concept from the slides, including feature branches, merging, rebasing, and branch pointers, becomes visible in real time. Work through as many levels as you can in the time available. The further you get, the deeper your mental model becomes.

---

## Hands On

Navigate to: **https://learngitbranching.js.org**

### Recommended Sequence

Work through the levels in order. The ones listed below align directly with what was covered in the slides today.

**Main: Introduction Sequence**
- *Introduction to Git Commits*: how commits build a history
- *Branching in Git*: creating and switching branches
- *Merging in Git*: combining branches with a merge commit
- *Rebase Introduction*: rewriting history onto another branch

**Main: Ramping Up**
- *Detach yo' HEAD*: understanding what HEAD points to
- *Relative Refs*: navigating history with `^` and `~`
- *Reversing Changes in Git*: `git reset` vs `git revert`

**Remote: Push & Pull**
- *Clone Intro*: what a remote repository looks like locally
- *Git Fetchin'*: downloading remote changes without merging
- *Git Pullin'*: fetch + merge in one step
- *Git Push*: sending local commits to a remote

> **Tip:** If you get stuck, type `hint` in the command window. If you want to reset a level, type `reset`.

---

### Things to Notice

As you work through the levels, pay attention to:

- How the **branch pointer** (the label on the commit graph) moves with each commit
- How a **merge commit** has two parent commits, while a rebase produces a linear history
- How `HEAD` is separate from the branch name and can be detached
- How `origin/main` (the remote tracking branch) does not move until you `fetch` or `pull`

These are the same mechanics your team uses every day. Most developers just cannot see them.

---

## Summary

After completing this lab you should be able to look at a `git log --graph` output and understand what you are seeing: which branches diverged, where they merged, and which commits belong to which history.

## Challenge

Complete the **Advanced Topics** section in the **Main** sequence:

- *Multiple Parents*: merging with specific parent selection
- *Branch Spaghetti*: complex cherry-pick and rebase scenarios

These levels are deliberately difficult. The goal is not to finish them. Get far enough that the commands stop feeling abstract.
